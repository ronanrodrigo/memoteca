# memoteca — Hermes Agent Project Context

> This file is auto-loaded by **Hermes Agent** when working in this repo (and any subdirectory up to the git root). It bridges memoteca conventions to Hermes-native concepts. The companion `AGENTS.md` (cross-agent portable) and `opencode.json` (OpenCode) remain intact — this file only adds Hermes-specific guidance.

## What this repo is

memoteca is a **GitHub template repository** that ships an agent orchestration pipeline (the harness) for scaffolding and shipping Next.js projects end-to-end. memoteca itself is NOT an app — it has no `package.json`, no `src/`, no Next.js project. It is the orchestration layer only. `make install/lint/build/test` will FAIL here — they require a scaffolded repo-project.

A repo-project created from this template via GitHub "Use this template" is initially byte-identical, then `make scaffold PROJECT_NAME="."` runs `create-next-app` in-place and adds the Next.js app. The pipeline runs end-to-end in the repo-project.

## Agent runtime — first-class support

Memoteca is agent-runtime-agnostic by design. The pipeline targets **Hermes Agent** and **OpenCode** as first-class runtimes. When you are running inside Hermes, prefer Hermes-native primitives:

| Concept | Hermes-native | OpenCode equivalent |
|---|---|---|
| Parallel/sub-agent | `delegate_task(goal=...)` (single) or `delegate_task(tasks=[...])` (batch up to 3) | `task` / `invoke` |
| Skills | `skill_view`, `skills_list`, `skill_manage`; skills live under `~/.hermes/skills/` or in-repo | `opencode.json` `skills.paths` |
| Project context | this `.hermes.md` (walks parents to git root) | `opencode.json` `instructions` |
| Persistent memory | `memory` tool (user + memory stores) | N/A |
| Scheduled jobs | `cronjob` tool | N/A |
| Terminal | `terminal` tool (bash via WSL on this Windows host) | `opencode` built-in |

### Sub-agent delegation (Hermes)

- Use `delegate_task(goal="...", context="...")` for a single focused subtask (e.g. "research 3 similar repos on GitHub", "write Jest tests for the auth module").
- Use `delegate_task(tasks=[{goal: ..., context: ...}, ...])` for up to 3 parallel independent workstreams. Children get isolated conversations and terminal sessions.
- Pass the issue number, file paths, and constraints in `context` — subagents have NO memory of your conversation.
- Subagent results return as a summary; intermediate tool output never enters your context.

### Skills (Hermes)

The 3 memoteca skills under `.memoteca/skills/` carry Hermes YAML frontmatter (`name: memoteca-*`) so they are installable into the Hermes skill tree:

- `memoteca-assistente` — the Assistant Work Loop, GitHub-native Mermaid, issue-as-source-of-truth, gcp/gpr shortcuts, worktree-by-feature. **This skill prevails on its topics.**
- `memoteca-intake` — intake flow (collect info, file issue, add to board).
- `memoteca-pr-visual-evidence` — PR visual evidence capture with Maestro CLI.

To install them into your Hermes profile, run from the repo root:

```
make hermes-setup
```

This symlinks (or copies on Windows) each `.memoteca/skills/<name>/` directory into `~/.hermes/skills/` so Hermes discovers them on next session. After installing, run `/reset` (or start a new `hermes` invocation) — the skill loader is cached per session.

Load a skill explicitly in-session with `/skill memoteca-assistente` or start Hermes with `hermes -s memoteca-assistente`.

## Pipeline — the source of truth

The authoritative pipeline spec lives in `AGENTS.md` and `.memoteca/agents/*.md`. This section is a quick orientation, not a redefinition.

```
Issue Created -> Orchestrator (Assistant Loop)
  -> Researcher -> Stack Selector
  -> [Human gate: post plan, wait for ok]
  -> Worktree (feature/<NN>-<short>)
  -> Implementer (scaffold + features)
  -> Validation -> Unit Tests -> Integration Tests
  -> PR (gcp | gpr | gcp and gpr)
  -> Deploy Preview + Validate HTTP 200
  -> Monitoring -> [Checks green + Preview OK] -> Merge
  -> Production deploy + Validate HTTP 200
  -> memory-finalize (closes issue)
```

All operations go through `make <target>`. Never invoke `gh`, `npm run`, `jest`, etc. directly. The GitHub issue is the source of truth — plan, memory, and decisions live as sequential comments on the issue (via `make memory-update`), NEVER as plan/MEMORY/TODO files in the repo.

## Conventions that matter to Hermes

- **First response in a new conversation starts with the emoji above** (Assistant Skill loading confirmation).
- **Human gate**: post the plan as a comment on the issue, wait for "ok". Only then start implementation. When "ok" is given, run `make memory-update ISSUE_NUMBER=<num> STATUS="Plan approved" COMMENT="Plan approved — starting implementation."`.
- **Worktree**: `feature/<NN>-<short>` off main. All work happens in the worktree.
- **Commits**: `<type>: <description> (#<NN>)`. `make gcp` auto-injects `(#NN)` from the branch name.
- **Mermaid**: GitHub-native ```mermaid blocks. No external viewer links.
- **Sub-agents**: `delegate_task` (Hermes) / `task` + `invoke` (OpenCode). Delegate early, in parallel.
- **No plan/memory files in the repo** — use the GitHub issue. This overrides any Hermes-default instinct to write `MEMORY.md` or `TODO.md`.

## Quick command reference

| What | Command |
|---|---|
| Install memoteca skills into Hermes | `make hermes-setup` |
| Query the central board for Todo items | `make tasks-listen` |
| Process a discovered issue | `make process-issue ISSUE_URL=<url>` |
| Scaffold the Next.js app (repo-project only) | `make scaffold PROJECT_NAME="."` |
| Update issue checkbox + board Status | `make memory-update ISSUE_NUMBER=<num> CHECKBOX="..." COMMENT="..."` |
| Finalize (check all + close issue) | `make memory-finalize ISSUE_NUMBER=<num>` |
| Commit + push | `make gcp MESSAGE="feat: ..."` |
| Open PR | `make gpr TITLE="feat: ..." BODY="..."` |
| Commit + push + PR | `make gcp-and-gpr MESSAGE="..." TITLE="..." BODY="..."` |
