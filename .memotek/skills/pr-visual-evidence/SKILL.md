# Skill: PR Visual Evidence

> Companion to the **Assistant** skill. Defines the workflow for capturing visual evidence (screenshots, screen recordings, videos) attached to Pull Requests.

## When to use

When Ronan requests visual evidence in a PR with words like: "screenshots", "screen recording", "simulator video", "visual proof", "evidence".

## Prerequisites

- Maestro CLI installed and in PATH.
- Target emulator/device/simulator available.
- Mobile app repo capable of building the target feature/PR version.

## Two capture methods

### Method 1 — Non-interactive point evidence (CLI)

Use when you want a very short flow (e.g., one tap + screenshot) and there's no need to integrate with an existing test.

```bash
maestro record --local <flow>.yaml
```

- Edit the `<flow>.yaml` to contain the minimal `launchApp`, `tapOn`, `assertVisible` that demonstrate the behavior.
- Run and record the execution; Maestro generates `.mp4` in the CWD.
- Host via GitHub (user-attachments) — never commit the binary to the repo.

### Method 2 — Video as integration test artifact

Use when the video is already part of a Maestro/CI integration test:

In the test's own `.yaml`, wrap the relevant steps with:

```yaml
- startRecording: evidence-feature-x
- tapOn: "Button X"
- assertVisible: "Result"
- stopRecording
```

- The `.mp4` file is produced as an artifact of the test itself (CI or local).
- In CI, configure the job to upload the artifact (`actions/upload-artifact` or equivalent).
- For PRs, upload manually via GitHub user-attachments in the comment.

## Media hosting — golden rule

- Media MUST be hosted by GitHub (comment attachments / user-attachments).
- Media must NEVER be committed to the repository.
- After hosting, edit the PR comment with the attachment URL generated by GitHub
  and markdown `![](<url>)` or link `[<description>](<url>)`.

## Complete workflow

1. Identify which method applies to Ronan's request.
2. (If Method 2) Ensure the target integration test exists; create/adjust the
   YAML by adding `startRecording`/`stopRecording` around the relevant steps.
3. Run the flow (Method 1) or the test (Method 2).
4. Host the `.mp4` on GitHub (user-attachments).
5. Edit the PR comment with:
   - Description of what is being demonstrated.
   - Link/markdown of the newly hosted media.
   - Source issue of the work (link).
6. Update the project's `MEMORY.md` with the PR comment URL (visual-evidence
   anchor) and check the corresponding checkbox in TODO.md.

## Anti-patterns

- ❌ Committing `.mp4`, `.png`, `.gif` to the repo.
- ❌ Hosting on external CDNs (imgur, dropbox) — use GitHub attachments.
- ❌ Attaching media without describing what it demonstrates.
- ❌ Skipping the evidence comment on the PR — the reviewer needs to see it inline.
